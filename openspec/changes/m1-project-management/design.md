## Context

项目管理系统的一期（M1），目标是为可视化爬虫系统建立配置和字段管理的基础能力。当前系统处于初始化阶段，backend和frontend基础工程已搭建完成，数据库已创建但schema尚未实现。

本设计覆盖：
- 后端：Entity、Repository、Service、Controller层的完整实现
- 前端：Vue Router配置、Pinia Store、API封装、页面组件
- 测试：Repository（Testcontainers）、Service（Mockito）、Controller（MockMvc）

## Goals / Non-Goals

**Goals:**
- 实现爬虫配置的完整CRUD（创建、查询、更新、删除）
- 实现字段的完整CRUD（作为配置的子资源）
- 配置删除时级联删除关联字段
- 前端实现配置列表页和配置编辑页
- **采用 TDD（测试驱动开发）模式开发**——先写测试再写实现，垂直切片

**Non-Goals:**
- 任务管理（crawl_task）
- 数据爬取功能
- 数据导出功能
- 字段排序
- 高级搜索/过滤
- 认证/鉴权

## Decisions

### 1. 字段作为配置的子资源（方案A）

**决策**: 字段通过 `/configs/:id/fields` 访问，不独立暴露于顶层路由

**理由**:
- 字段必须关联配置存在，无独立意义
- 简化API复杂度，避免孤立的字段资源
- 与领域模型语义一致

**替代方案**:
- 方案B：字段独立资源，通过 `configId` 关联（被否决，语义不清晰）

### 2. 更新配置时字段全量替换

**决策**: PUT `/configs/:id` 时，body中的fields数组会整体替换现有字段

**理由**:
- 简化前端实现，无需计算diff
- 符合"配置快照"语义
- 避免部分更新的边界情况

### 3. 页面类型枚举

**决策**: pageType使用枚举 `LIST_DETAIL`, `DETAIL_ONLY`

| 枚举值 | 说明 |
|--------|------|
| LIST_DETAIL | 列表页+详情页模式，需要detail_url字段 |
| DETAIL_ONLY | 仅详情页模式 |

**理由**: 覆盖两种主流爬取场景，枚举比字符串更类型安全

### 4. 选择器类型枚举

**决策**: selectorType使用枚举 `CSS`, `XPATH`

**理由**: MVP同时支持CSS和XPATH选择器，枚举约束类型

### 5. 状态默认值为STOPPED

**决策**: 新建配置的status默认为STOPPED

**理由**: 新建配置不会自动启动爬取，需用户明确操作

### 6. 前端页面路由设计

**决策**: `/configs` 列表页、`/configs/new` 新建页、`/configs/:id` 编辑详情合一页

**理由**:
- 详情页和编辑页合并减少重复代码
- 新建和编辑使用同一组件，通过id参数区分

### 7. 后端分层架构

**决策**: Controller -> Service -> Repository -> Entity

**理由**:
- Spring Boot标准分层
- 清晰职责分离
- 便于单元测试

### 8. Testcontainers用于Repository测试

**决策**: Repository集成测试使用Testcontainers启动真实PostgreSQL容器

**理由**:
- 真实数据库环境，测试更可靠
- CI环境可重复运行
- 避免内存数据库与真实数据库行为差异

### 9. 采用 TDD（测试驱动开发）模式

**决策**: 全程采用 TDD 红绿循环开发——先写失败的测试，再写最小代码让测试通过，最后重构。严格遵循垂直切片（Vertical Slicing），不做水平切片。

**理由**:
- 测试作为可执行的规范，验证行为而非实现细节
- 早期暴露设计问题，避免过度设计
- 重构信心：测试不耦合实现细节，重构时不会大量失效
- 防止"先写完所有代码再补测试"导致的测试与实现脱节、测试变成验证实现而非验证行为

**TDD 核心规则**:
- **RED**: 先写一个测试，测试必须失败（因为实现还不存在）
- **GREEN**: 写最小代码让测试通过，禁止添加超出当前测试的功能
- **REFACTOR**: 测试保持绿色的前提下，重构代码消除重复、改善设计
- **垂直切片**: 一次一个行为的完整循环（测试+实现），不做"先写完所有测试再写所有实现"的水平切片
- **测试公共接口**: 测试只通过公共 API 验证行为，不验证内部实现
- **最小可行**: 不为不存在的需求写测试，不为推测的功能写实现

**禁止的反模式**:
- 先写完所有测试再写所有实现（水平切片）
- 为私有方法写测试
- 过度 mock 内部协作者
- 写完代码后补测试（事后测试，非 TDD）
- 重构时让测试处于 RED 状态

**TDD 应用范围**:
- 后端所有业务逻辑（Service 层）
- 后端 Controller 层（通过 MockMvc）
- 后端 Repository 层（通过 Testcontainers 集成测试）
- 前端 Pinia Store 和 API 模块的关键行为
- 前端页面组件的渲染逻辑和交互行为

**TDD 不适用场景**:
- 纯 Entity/POJO/枚举定义（无可测试行为）
- 配置文件、纯静态类
- 一次性的脚手架代码

## Risks / Trade-offs

| 风险 | 描述 | 缓解措施 |
|------|------|----------|
| 字段全量替换丢失编辑中间状态 | 用户编辑字段时页面刷新会导致已修改但未保存的字段丢失 | 前端页面刷新时从API重新加载最新配置 |
| Testcontainers启动慢 | 首次启动Docker容器可能需要较长时间 | 测试前确保Docker Desktop已启动，CI配置合适的超时时间 |
| 前后端字段校验不一致 | 前端表单校验与后端DTO校验可能出现差异 | 定义统一的校验规则文档，API层作为唯一数据校验点 |
| TDD 初期节奏较慢 | 先写测试会让初期开发速度感觉变慢 | 短期投入换取长期可维护性和重构信心；随着熟练度提升会显著加快 |
