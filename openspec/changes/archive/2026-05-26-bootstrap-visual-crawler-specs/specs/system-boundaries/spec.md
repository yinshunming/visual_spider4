# system-boundaries（系统边界）

## ADDED Requirements

### Requirement: 单一浏览器会话（MVP 约束）
系统 SHALL 仅支持同一时间一个活动浏览器会话。这是 MVP 加诸的约束，非底层 Playwright 引擎的技术限制。

#### 场景：在已有会话时尝试打开第二个会话
- **WHEN** 浏览器会话已活动，用户请求打开另一个页面
- **THEN** 系统关闭已有页面，在同一会话中打开新 URL（单会话复用）；不报错

### Requirement: 单用户访问（MVP 约束）
系统 SHALL 设计为单一并发用户使用。MVP 中无用户认证、会话隔离或权限模型。所有 API 调用操作共享数据。

#### 场景：两用户同时操作
- **WHEN** 两用户同时使用系统
- **THEN** 两用户的操作影响相同的共享数据，无隔离；这是预期的 MVP 行为

### Requirement: 无认证或授权
系统 SHALL 在 MVP 中不实现任何认证、授权或用户身份机制。

#### 场景：无凭证 API 访问
- **WHEN** 任意客户端向系统发送任何 API 请求
- **THEN** 系统处理请求，不要求凭证

### Requirement: 无反爬取对抗措施
系统 SHALL 不实现任何反爬取策略（User-Agent 轮换、请求限速、代理轮换、验证码处理或 Cookie/会话管理）。

#### 场景：目标网站阻止请求
- **WHEN** 目标网站阻止或限速爬虫请求
- **THEN** 系统无内置缓解措施；爬取失败，错误信息记录在 error_message 中

### Requirement: 不支持定时或循环爬取
系统 SHALL 在 MVP 中不支持 cron、定时器或事件驱动的自动爬取触发。所有爬取由用户手动启动。

#### 场景：尝试设置定时爬取
- **WHEN** 用户尝试配置定时爬取
- **THEN** 系统不支持该功能；属于 MVP 范围外

### Requirement: 不支持增量爬取
系统 SHALL 不跟踪哪些项目之前已爬取过。对同一配置启动新任务会从头重新爬取所有项目。

#### 场景：重新运行同一配置的爬取
- **WHEN** 对已爬取过的配置启动任务
- **THEN** 系统创建新的 list_page、list_item 和 article 记录；之前任务的记录保持不变

### Requirement: 资源限制
系统 SHALL 对单个任务内的并发详情页提取数量实行硬限制。MVP 默认并发为 1（顺序提取）。

#### 场景：顺序提取
- **WHEN** 包含 10 个详情 URL 的任务以并发数 = 1 运行
- **THEN** URL 逐个处理

### Requirement: 浏览器崩溃时的优雅降级
如果 Playwright 浏览器进程意外崩溃，系统 SHALL 捕获错误，将其记录在受影响实体的 error_message 中，并允许任务继续或停止。

#### 场景：提取期间浏览器崩溃
- **WHEN** 提取详情页时浏览器进程意外终止
- **THEN** article 保存时状态为 FAILED，error_message 描述崩溃情况；如停止标志未设置，爬取引擎 MAY 尝试重启浏览器并继续处理下一项目

### Requirement: 选择器模式一致性
配置 SHALL 以单一选择器模式（CSS 或 XPATH）运行。不可能在单个配置中同时使用两种选择器类型。

#### 场景：切换选择器类型使已有选择器失效
- **WHEN** 用户将配置的选择器类型从 CSS 改为 XPATH
- **THEN** 所有之前生成的选择器（基于 CSS）在新模式下失效；用户必须重新生成或手动更新选择器

## 开放问题与风险

| 项目 | 问题 | 风险 |
|------|------|------|
| 会话恢复 | 如果后端进程在爬取中途重启，内存中的浏览器会话丢失。这对 MVP 可接受吗，还是应该持久化会话状态？ | 低 — MVP 假设后端进程稳定 |
| 内存压力 | 大页面（如 1000 项的商品列表页）的 raw_html 可能达数十 MB。有存储/传输限制吗？ | 中 — 未定义明确限制，需监控 |
| 并发扩展 | MVP 为单并发；后续迭代需要有可配置并发（会话池）的明确扩展点 | 低 — 已知扩展项已记录 |
